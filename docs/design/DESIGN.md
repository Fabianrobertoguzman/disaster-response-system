## Architectural decisions (rationale)

- **Three tiers with a shared protocol.** The client never touches the database;
  it speaks only to `ServerStub`, which exchanges `Serializable` `Request`/
  `Response` objects (with explicit `serialVersionUID`) defined once in
  `edu.cqu.drs.protocol` and depended on by both sides, so the tiers never drift.
- **MVP on the client, Service layer on the server.** FXML views + presenters run
  on the client; the A2 presenters' business logic moved to server-side services
  (`IncidentService`). The server is a *Service/Application* layer, never "the
  Presenter".
- **Concurrency (V-C1).** `DrsServer` accepts connections and runs one pooled
  `ClientHandler` per client. Shared state is reached only through the thread-safe
  data tier: the JDBC DAOs open a fresh connection per request (a JDBC
  `Connection` is not thread-safe), and the in-memory test backend guards its
  compound operation with a `ReentrantReadWriteLock` and hands out defensive
  copies. The object-stream handshake creates and flushes the `ObjectOutputStream`
  before the `ObjectInputStream` on both ends to avoid the classic deadlock.
- **Database-generated IDs (§2.4).** Every table has a `BIGINT AUTO_INCREMENT`
  surrogate primary key generated by MySQL, plus a `CHAR(36) uuid` column carrying
  the domain identity; the DAOs map between them.
- **Security (§2.5), with the V-C4 distinction.** *Access rights*: PBKDF2 password
  **hashing** (one-way) + role-gated actions. *Encryption/decryption*: genuine
  **reversible** AES-256-GCM (`FieldCipher`) on the incident description at rest —
  hashing is explicitly **not** counted as this measure. *Time stamping*:
  server-assigned audit timestamps. *Non-repudiation*: an append-only audit trail
  of actor + action + time.

## Known limitations (honest scope)

- **`incidents.reported_by`** is provisioned in the schema and the ERD as the
  reporter foreign key, but `IncidentDaoImpl.insert` does not yet populate it, so
  it is currently always `NULL`. The acting user *is* recorded against the action
  in the audit trail; wiring the reporter FK is a small follow-up.
- **`notifications` and `partner_agencies`** are created and (for agencies) seeded
  in the schema — carried forward from the A1/A2 partner-notification concept —
  but are not yet wired to a DAO at runtime in this build.
- **The two new features (f1, f2)** are specified (R5) and shown in D2 but are not
  implemented in this increment.

## Reuse disposition

- **A1 (carried/extended):** requirements catalogue and use-case framing.
- **A2 (extended):** the domain model, the MVP structure, the rule-driven alert
  recommender, the pom and test conventions.
- **New (0% reuse):** the distributed architecture (D1), the protocol module, the
  multi-threaded server, the MySQL data tier and ERD (D5), and the §2.5 security
  tier; plus the two planned features.
